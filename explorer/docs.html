Chakram Protocol Documentation

Chakram (CHK) is a Kerala-inspired UTXO cryptocurrency using RandomX proof-of-work, Ed25519 signatures, and a lightweight JSON HTTP API. This document is the complete reference for wallets, exchanges, and node operators.

All API values are denominated in Cash — the smallest indivisible unit. Display amounts in CHK by dividing by 1,000,000.

HTTP API Reference

The Chakram node exposes a JSON REST API over HTTP. Every response is application/json. Errors return {"error":"message"} with an appropriate HTTP status code.

Base URL: http://<node-ip>:8339 (mainnet) · http://<node-ip>:18339 (testnet)
CORS is unrestricted — all origins are allowed.

POST /tx/submit
Broadcasts a signed transaction to the network. The node validates the transaction, adds it to the mempool, and relays it to all connected peers. See the Transaction Protocol section for how to construct a valid transaction.
POST /tx/submit
Content-Type: application/json

Encoding: All []byte fields (TxID, PublicKey, Signature, PublicKeyHash) must be base64-encoded (standard encoding). This matches Go's default JSON marshalling for byte slices.

Note: the /tx/{txid} lookup endpoint returns TxIDs in hex. You must convert to base64 when building transaction inputs.

Keys & Addresses

Key Generation

Chakram uses Ed25519 for all signing. The private key is a 32-byte seed; the public key is derived deterministically.

// 1. Generate 16 bytes of random entropy (BIP39 uses 128-bit entropy)
entropy = random_bytes(16)

// 2. Derive seed from entropy (matches BIP39 mnemonic → seed path)
seed = SHA256(entropy)          // 32 bytes — this is the Ed25519 private key seed

// 3. Derive public key
publicKey = ed25519.PublicKey(seed)   // 32 bytes
The seed (SHA256 of entropy) is equivalent to a private key. Never expose or transmit it. Store only the encrypted form.

Mnemonic Recovery Phrase

Wallet recovery phrases follow the BIP39 standard (English, 2048-word list) with 128-bit entropy encoded as 12 words.

// Encoding:
entropy (16 bytes) → SHA256 → take top 4 bits as checksum
pack (entropy ++ checksum) into 132 bits → 12 × 11-bit BIP39 word indices

// Recovering seed from mnemonic:
words → indices → 128 bits entropy + 4-bit checksum
verify checksum
seed = SHA256(entropy)

Address Format

A Chakram address encodes the public key hash — the 20-byte RIPEMD160(SHA256(publicKey)) — with a 4-byte checksum, using Base58Check with a CK1 prefix.

// Step-by-step:
pkh      = RIPEMD160(SHA256(publicKey))       // 20 bytes
checksum = SHA256(SHA256(pkh))[0:4]           // first 4 bytes
payload  = pkh ++ checksum                    // 24 bytes
address  = "CK1" + Base58Encode(payload)

// Validation:
raw      = Base58Decode(address[3:])          // strip "CK1"
pkh      = raw[0:20]
cs       = raw[20:24]
assert SHA256(SHA256(pkh))[0:4] == cs        // checksum must match

The Base58 alphabet used is the standard Bitcoin alphabet: 123456789ABCDEFGHJKLMNPQRSTUVWXYZabcdefghijkmnopqrstuvwxyz

Transactions

Transaction Structure

A Chakram transaction spends one or more UTXOs (inputs) and creates one or more new outputs. All byte fields in the JSON are base64-encoded.

FieldTypeDescription
TxIDbase64 (32 bytes)Transaction identifier — computed, not chosen
IsCoinbaseboolAlways false for user transactions
InputsarrayUTXOs being spent
OutputsarrayNew UTXOs being created
Timestampint64 (Unix seconds)Current time at signing

Input fields

FieldTypeDescription
TxIDbase64 (32 bytes)The UTXO's originating transaction ID
OutputIndexuint32Index within that transaction's outputs
PublicKeybase64 (32 bytes)Spender's Ed25519 public key
Signaturebase64 (64 bytes)Ed25519 signature over the input message

Output fields

FieldTypeDescription
Valueuint64Amount in Cash (must be > 0)
PublicKeyHashbase64 (20 bytes)Recipient's RIPEMD160(SHA256(pubkey))

Transaction ID Computation

The TxID is SHA256(SHA256(canonical_bytes)) where the canonical serialization is:

// Canonical byte layout (little-endian integers):
[1 byte]   IsCoinbase flag (0x00 for user transactions)
for each input:
  [32 bytes] TxID (raw bytes, from base64)
  [4 bytes]  OutputIndex (uint32, little-endian)
for each output:
  [8 bytes]  Value (uint64, little-endian)
  [20 bytes] PublicKeyHash (raw bytes, from base64)
[8 bytes]  Timestamp (int64, little-endian)

TxID = SHA256(SHA256(canonical_bytes))    // 32 bytes
Store as base64 in the transaction JSON.
Compute the TxID before signing. The canonical bytes use empty (zero-filled) Signature and PublicKey fields. The TxID is stable and does not change when you add signatures.

Signing

Each input in a transaction must be individually signed. The signature proves ownership of the UTXO being spent.

// For each input i:
txid_bytes    = base64_decode(input[i].TxID)      // 32 bytes
index_bytes   = uint32_to_bytes_LE(input[i].OutputIndex)  // 4 bytes
preimage      = txid_bytes ++ index_bytes          // 36 bytes
message       = SHA256(preimage)                  // 32 bytes

signature     = ed25519.Sign(seed, message)        // 64 bytes

input[i].Signature = base64_encode(signature)
input[i].PublicKey = base64_encode(ed25519.PublicKey(seed))

Verification (node-side)

The node verifies each input by:

  1. Looking up the UTXO — it must exist and not be spent
  2. Checking coinbase maturity — coinbase UTXOs need 10 confirmations
  3. Verifying the Ed25519 signature against SHA256(TxID‖OutputIndex)
  4. Checking that RIPEMD160(SHA256(publicKey)) matches the UTXO's stored pubkey hash

Fee

The network enforces a minimum fee of 1,000 Cash (0.001 CHK). Fee is implicit: fee = sum(input values) - sum(output values). Add a change output back to yourself if the difference would be too large.

total_in   = sum of all input UTXO values
total_out  = sum of all output values
fee        = total_in - total_out
assert fee >= 1000  // minimum fee in Cash

UTXO Model

Chakram uses an Unspent Transaction Output (UTXO) model. A wallet's balance is the sum of all UTXOs whose PublicKeyHash matches the wallet's address. There are no account balances — only outputs.

// Typical transfer flow:

1. Query GET /utxos/CK1... → list of spendable outputs
2. Select UTXOs whose total value >= amount + fee (coin selection)
3. Build transaction:
   - Inputs:  selected UTXOs
   - Outputs: [recipient output, optional change output back to sender]
   - Timestamp: current Unix time
4. Compute TxID
5. Sign each input
6. POST /tx/submit

// Change output:
change = total_input - amount - fee
if change > 0:
    add output: { Value: change, PublicKeyHash: sender_pkh }
Only select UTXOs where mature === true. Block rewards (coinbase outputs) require 10 confirmations (~5 minutes) before they can be spent. Regular transaction outputs have no maturity requirement.

Running a Node

Binary

Chakram is a single Go binary. No dependencies or configuration files are required at runtime.

# Download the latest binary from the releases page
chmod +x chakram

# Run a full node (mainnet)
./chakram

# Run on testnet
./chakram --testnet

# Connect to a specific peer
./chakram --peer 35.207.229.32:18338

# Custom data directory
./chakram --datadir /var/lib/chakram

# Custom RPC port
./chakram --rpc-port 8339

Data Directory

By default, the node stores all blockchain data in ./chakram-data/ (current directory). The database is BadgerDB — do not copy or modify it while the node is running.

Ports to Open

PortProtocolPurpose
8338 (mainnet) / 18338 (testnet)TCPP2P — peer connections
8339 (mainnet) / 18339 (testnet)TCPHTTP — RPC, explorer, wallet
For a public seed node, open both ports in your firewall. For a private node or wallet server, only the RPC port needs to be reachable from external clients.

Mining

Chakram uses RandomX proof-of-work — an ASIC-resistant algorithm designed for CPU mining. Mining is built into the node binary; no separate miner software is needed.

# Enable mining (block rewards sent to the node's wallet address)
./chakram --mine

# Mine on testnet
./chakram --testnet --mine
ParameterValue
AlgorithmRandomX (light mode)
Epoch length64 blocks (~32 minutes)
Mining threads1 (configured in binary)
Block reward (era 1)50 CHK
Halving intervalEvery 2,102,400 blocks (~2 years)

The RandomX epoch seed is deterministic: "chakram-epoch-N" where N is height / 64. All nodes compute the same seed for the same epoch — no consensus on seed data is required.

P2P Protocol

The Chakram P2P protocol uses a custom binary framing over TCP. Each message is a length-prefixed frame with a 4-byte magic number header.

Message Frame

[4 bytes]  Magic (0x43484143 mainnet / 0x43484154 testnet)
[1 byte]   Message type
[4 bytes]  Payload length (uint32, big-endian)
[N bytes]  Payload (JSON-encoded)

Message Types

TypeNameDirectionDescription
0x01VersionBothHandshake — announces height and version
0x02GetBlocksBothRequest blocks from height N, up to count C
0x03BlocksBothResponse carrying one or more blocks
0x04InvBothAnnounce new block or transaction hash
0x05GetDataBothRequest a specific block or tx by hash
0x06TxBothRelay a transaction
0x07GetPeersBothRequest peer list
0x08PeersBothResponse with known peer addresses
The P2P wire protocol is intended for full node implementations. Wallets and exchanges only need the HTTP API — they do not need to speak the P2P protocol.

Integration Guide

Exchange Integration

To add CHK deposit/withdrawal support to an exchange:

  1. Run a Chakram full node and keep it synced
  2. Generate a unique deposit address per user using the key derivation above
  3. Poll GET /address/{address} to detect balance changes, or scan blocks with GET /block/{height}
  4. To credit deposits, verify the UTXO appears in GET /utxos/{address} and has mature: true
  5. To process withdrawals, build and sign a transaction offline, submit via POST /tx/submit

Watch-Only Wallet

A watch-only wallet only needs the address (no private key). Use GET /address/{addr} for balance and GET /utxos/{addr} to monitor incoming payments.

Signing in JavaScript (Browser / Node.js)

// Using @noble/ed25519 v2 and @noble/hashes (no WebCrypto required)
import { etc, getPublicKey, sign } from '@noble/ed25519';
import { sha256 } from '@noble/hashes/sha256';
import { sha512 } from '@noble/hashes/sha512';
import { ripemd160 } from '@noble/hashes/ripemd160';
etc.sha512Sync = sha512;

// Derive address from seed
const pub = getPublicKey(seed);                         // 32 bytes
const pkh = ripemd160(sha256(pub));                     // 20 bytes

// Sign an input
const preimage = new Uint8Array([...txid_bytes, ...index_le_4bytes]);
const message  = sha256(preimage);                      // 32 bytes
const sig      = sign(message, seed);                   // 64 bytes

Signing in Python

from cryptography.hazmat.primitives.asymmetric.ed25519 import Ed25519PrivateKey
import hashlib, struct

seed = hashlib.sha256(entropy).digest()
privkey = Ed25519PrivateKey.from_private_bytes(seed)
pubkey  = privkey.public_key().public_bytes_raw()       # 32 bytes

# Sign input i
preimage = txid_bytes + struct.pack('<I', output_index)
message  = hashlib.sha256(preimage).digest()
sig      = privkey.sign(message)                        # 64 bytes

Signing in Go

import (
    "crypto/ed25519"
    "crypto/sha256"
    "encoding/binary"
)

// Derive key
seed := sha256.Sum256(entropy)
privKey := ed25519.NewKeyFromSeed(seed[:])
pubKey  := privKey.Public().(ed25519.PublicKey)

// Sign input
preimage := append(txID, binary.LittleEndian.AppendUint32(nil, outputIndex)...)
message  := sha256.Sum256(preimage)
sig      := ed25519.Sign(privKey, message[:])
